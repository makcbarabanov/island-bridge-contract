# Общие данные БД (Остров + бот)

Цель: один источник правды для пользователя и между вебом и ботом.

## Сейчас

- Пользователи и мечты — таблицы Острова (см. `tables.md` в репо Острова).
- Бот дублирует часть «памяти» в файлах `user_profiles/` на сервере с Bloom — **временно**; долгосрочно перенос в БД Острова (отдельное проектирование).

## Планируется

- `users.telegram_id` — связка Telegram ↔ аккаунт.
- `daily_report_log` (или аналог) — антидубли вечернего отчёта.
- Таблицы/поля для biography / хвост чата / bloom_traits — **TBD**, согласовать здесь после миграции.

## Bridge: _educ_ контур (внедрено)

Источник: репозиторий `bridge`, SQL в `docs/handbook/marathon_educ_schema.sql` и `docs/handbook/marathon_educ_seed_from_legacy.sql`.

### Новые таблицы

- `_educ_reports_raw` — сырые события отчётов из Telegram/Web/legacy, включая версионность edits и payload классификатора.
- `_educ_reports_daily` — факт сдачи отчёта по `(user_id, report_date)` с источником (`telegram`/`web`/`mixed`) и статусом.
- `_educ_manifest_items` — нормализованные пункты манифеста по пользователю и месяцу марафона.
- `_educ_report_matches` — сопоставление фрагментов отчёта с пунктами манифеста (тип совпадения, confidence, review-флаг).
- `_educ_review_queue` — очередь сомнений/вопросов для дообучения и ручной валидации.
- `_educ_daily_snapshots` — ежедневные снимки статистики (active/reported/missing) для аудита динамики.
- `_educ_user_patterns` — персональные паттерны пользователя (сокращения, эмодзи, маркеры статуса/даты).

### Новое представление

- `active_marathon_users` — активные участники периода 1–21:
  - активен, если есть манифест или отчёт в периоде;
  - учитывает как legacy (`_old_old_manifest`, `_old_old_reports`), так и новый `_educ_*` контур;
  - если в текущем месяце данных нет, использует последний месяц с данными (fallback).

### Бизнес-правила (текущая фиксация)

- SSOT: для марафона приоритет хранилища — БД.
- Часовой пояс расчётов отчётного дня: `Europe/Moscow`.
- Конфликт источников: если `web=выполнено`, `tg=не найдено` — считаем выполненным с `source=web`.
- Пограничные случаи классификации отправляются в `_educ_review_queue` для ручного решения и обучения правил.
